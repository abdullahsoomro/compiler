§ 1 — Foundations

What is Parsing?

Parsing (syntactic analysis) is the second phase of a compiler. It receives a flat stream of tokens from the lexer and verifies that they form a grammatically valid sentence, producing a parse tree or abstract syntax tree (AST) as output.

// Compiler pipeline Source Code → Lexer (tokens) → Parser (AST) → Semantic Analysis → Code Gen

§ 2 — The Grammar & LR Concepts

The Grammar Used Throughout

(0)S' → S
(1)S → Aa
(2)S → bAc
(3)S → dc
(4)S → bda
(5)A → d

S' is the augmented start symbol added to signal acceptance. Terminals: a b c d $. Non-terminals: S' S A.

LR Parser Family

ParserLookaheadItem TypePower
LR(0)NoneLR(0) itemsWeakest
SLR(1)FOLLOW sets (global)LR(0) items↑
LALR(1)Merged LR(1) lookaheadsLR(1) merged↑↑
CLR(1)Exact per-item lookaheadsLR(1) itemsStrongest

Closure and GOTO

Closure(I): For any item with dot before a non-terminal B (i.e., A → α • B β), add all items B → • γ for every production B → γ. Repeat until stable.

GOTO(I, X): Move the dot past symbol X in every eligible item of I, then take closure. Represents the next state after recognising X.

Key concept

An LR(0) item is a production with a dot (•) marking progress. An LR(1) item adds a lookahead token in brackets: [A → α • β, a] — reduce by A→α only when next input is a.

§ 4 — SLR(1)

SLR(1): FIRST & FOLLOW Sets

SLR uses the same 11 LR(0) states but reduces only when the next token is in FOLLOW(A) for the lhs A.

FIRST Sets

FIRST(a){ a }
FIRST(b){ b }
FIRST(c){ c }
FIRST(d){ d }
FIRST(A){ d }A → d only
FIRST(S){ d, b }S→Aa: FIRST(A)={d}; S→bAc, S→bda: {b}; S→dc: {d}
FIRST(S'){ d, b }= FIRST(S)

FOLLOW Sets

FOLLOW(S'){ $ }
FOLLOW(S){ $ }S'→S, so FOLLOW(S) ⊇ FOLLOW(S')
FOLLOW(A){ a, c }S→Aa: 'a' follows A · S→bAc: 'c' follows A

Conflict at (State 4, 'c')

In I₄: S → d • c wants to shift 'c'. A → d • wants to reduce (r5) for all tokens in FOLLOW(A) = {a, c}. Both apply to 'c' — a Shift-Reduce conflict. SLR uses FOLLOW globally and cannot distinguish that A here only feeds S→Aa (where only 'a' follows). This grammar is not SLR(1).

§ 5 — CLR(1)

Canonical LR(1) Parsing

CLR(1) items carry exact per-derivation lookaheads: [A → α • β, a]. Lookaheads are computed as FIRST(β · current_lookahead) — far more precise than global FOLLOW sets.

LR(1) States — Key States

C₀ START
[S' → • S, $]
[S → • Aa, $]
[S → • bAc, $]

A→C₆d→C₇

[S → d • c, $] → shift on 'c'
[A → d •, a] → reduce on 'a' only
c → C₈
C₇ — GOTO(C₃, d)
[S → bd • a, $] → shift on 'a'
[A → d •, c] → reduce on 'c' only
a → C₁₀
Why CLR(1) has no conflicts

In C₄: A → d • has lookahead {a} (A only feeds S→Aa from C₀). S→d•c shifts on 'c'. Different tokens — zero conflict. In C₇: A→d• has lookahead {c} (A feeds S→bAc from C₃), S→bd•a shifts on 'a'. Again, no overlap.

§ 6 — LALR(1)

LALR(1): State Merging

LALR(1) merges CLR(1) states that share the same core (identical dotted productions, ignoring lookaheads), then unions their lookaheads. This keeps the state count equal to LR(0)/SLR while retaining some lookahead precision.

This grammar

All CLR(1) states have unique cores — no two states share the same set of dotted productions. Therefore no merging occurs, and the LALR(1) table is identical to the CLR(1) table. This grammar is LALR(1).

When Merging Causes Conflicts

If two CLR(1) states share a core but have disjoint lookaheads, merging unions them. If the merged lookaheads now overlap with another reduce or shift action, a Reduce-Reduce conflict appears — one that didn't exist in CLR(1). This is the sole cost of LALR(1) over CLR(1).

// Hypothetical: two CLR states with same core, different lookaheads
State X: [A → α •, a] // reduce A→α on 'a'
State Y: [A → α •, b] // reduce A→α on 'b'

Merged XY: [A → α •, a/b] // fine unless...
If XY also has [B → β •, a] → Reduce-Reduce conflict on 'a'!

Practical hierarchy reminder

LR(0) ⊂ SLR(1) ⊂ LALR(1) ⊂ CLR(1)

yacc / bison implement LALR(1) — the industry standard.
CLR(1) tables can be exponentially larger for some grammars.
